# 점심 모집 앱 - 비즈니스 규칙

---

## 1. 모집 규칙

### 1-1. 모집글 작성

| 규칙 | 상세 |
|------|------|
| 작성 권한 | 워크스페이스 멤버 누구나 작성 가능 |
| 같은 날 여러 모집 | **가능**. 한 사람이 같은 날 여러 개의 모집글을 올릴 수 있다 (예: 점심, 간식) |
| 미래 날짜 모집 | **가능**. 내일, 모레 등 미래 날짜도 모집 가능 |
| 과거 날짜 모집 | **불가**. 과거 날짜는 선택할 수 없다 |
| 오늘이지만 지난 시간 | **경고 후 허용하지 않음**. 현재 시간 이전의 시간은 선택 불가 |
| 필수 입력 | 메뉴명, 날짜, 시간, 최대인원 |
| 선택 입력 | 장소, 메모 |
| 최대인원 범위 | 2~10명 |
| 모집자 자동 참여 | 모집글 작성 시 모집자는 **자동으로 첫 번째 참여자**가 된다 |

### 1-2. 모집글 수정

| 규칙 | 상세 |
|------|------|
| 수정 권한 | **모집자 본인만** 수정 가능 |
| 마감 후 수정 | **불가** |
| 최대인원 축소 | 현재 참여인원 미만으로 줄일 수 없다 (예: 3명 참여 중이면 최소 3명) |
| 날짜/시간 변경 | **가능**. 단, 과거 날짜/시간으로 변경 불가 |
| 수정 알림 | 참여자들에게 수정 알림 표시 (v2. MVP에서는 알림 없음) |

### 1-3. 모집글 삭제

| 규칙 | 상세 |
|------|------|
| 삭제 권한 | **모집자 본인만** 삭제 가능 |
| 참여자가 있는 상태에서 삭제 | **가능**. 확인 다이얼로그에서 "참여자들의 참여가 모두 취소됩니다" 경고 |
| 마감된 모집글 삭제 | **가능**. 확인 다이얼로그 표시 |
| 삭제 후 복구 | **불가**. 소프트 딜리트로 DB에는 남기되 사용자에게는 보이지 않음 |

### 1-4. 모집 마감

| 규칙 | 상세 |
|------|------|
| 자동 마감 (시간) | 설정한 모집 시간이 되면 자동으로 마감된다 (예: 12:00 모집이면 12:00에 마감) |
| 자동 마감 (인원) | 최대인원이 충족되면 즉시 자동 마감된다 |
| 수동 마감 | 모집자가 "마감하기" 버튼으로 언제든 수동 마감 가능 |
| 마감 해제 | **불가** (MVP). 마감된 모집은 되돌릴 수 없다 |
| 마감 후 상태 | 모집글은 "마감" 상태로 남아있고, 참여자 목록도 유지된다 |

## 2. 참여 규칙

### 2-1. 참여

| 규칙 | 상세 |
|------|------|
| 참여 조건 | 워크스페이스 멤버 누구나 참여 가능 |
| 같은 날 여러 모집 참여 | **가능**. 제한 없음 (본인이 관리) |
| 같은 모집 중복 참여 | **불가**. 이미 참여 중이면 "참여하기" 버튼 대신 "참여 취소" 표시 |
| 마감된 모집 참여 | **불가** |
| 인원 꽉 찬 모집 참여 | **불가**. "인원이 마감되었습니다" 메시지 |

### 2-2. 참여 취소

| 규칙 | 상세 |
|------|------|
| 취소 가능 시점 | **마감 전까지만** 취소 가능 |
| 마감 후 취소 | **불가**. "마감된 모집은 취소할 수 없습니다" 메시지 |
| 모집자의 참여 취소 | **불가**. 모집자는 자신의 모집에서 빠질 수 없다. 모집 자체를 삭제해야 한다 |
| 취소 후 재참여 | **가능**. 인원에 빈자리가 있으면 다시 참여 가능 |

### 2-3. 동시성 처리

| 상황 | 처리 방식 |
|------|----------|
| 동시에 여러 명이 참여 시도 (인원 1자리 남음) | 서버에서 트랜잭션으로 처리. 먼저 요청 도착한 사람만 참여 성공. 나머지는 "인원이 마감되었습니다" |
| 마감 직전 참여 시도 | 서버 시간 기준으로 마감 여부 판단. 마감 후 도착한 요청은 거부 |

## 3. 워크스페이스 규칙

### 3-1. 생성

| 규칙 | 상세 |
|------|------|
| 생성 조건 | 누구나 생성 가능 (로그인 없이) |
| 생성 시 발급 | 초대 링크 (멤버용) + 관리 링크 (생성자용) |
| 워크스페이스 이름 | 2~30자. 중복 허용 (slug로 구분) |
| slug 생성 | 이름 기반 자동 생성 + 랜덤 suffix (예: "abc-tech-dev-a1b2") |

### 3-2. 멤버

| 규칙 | 상세 |
|------|------|
| 참여 방식 | **초대 링크로만** 참여 가능 (공개 검색 없음) |
| 멤버 상한 | **50명** (MVP). 초과 시 "워크스페이스 멤버 수가 초과되었습니다" |
| 역할 구분 | **없음** (MVP). 생성자만 관리 링크를 통해 설정 변경 가능 |
| 닉네임 중복 | **허용**. 내부적으로는 고유 토큰으로 구분 |
| 닉네임 변경 | 가능. 기존 모집글/참여 기록에도 새 닉네임 반영 |

### 3-3. 초대 링크

| 규칙 | 상세 |
|------|------|
| 링크 형식 | `/{workspace-slug}/invite/{invite-code}` |
| 유효 기간 | **무기한** (재발급 전까지) |
| 재발급 | 관리자(생성자)만 가능. 재발급 시 기존 링크 즉시 무효화 |
| 동시 유효 링크 | 1개만 유효 |

### 3-4. 탈퇴

| 규칙 | 상세 |
|------|------|
| 일반 멤버 탈퇴 | 설정에서 "워크스페이스 나가기" 클릭. 브라우저에서 해당 토큰 삭제 |
| 탈퇴 후 데이터 | 기존 모집글과 참여 기록은 닉네임으로 남아있다 (삭제하지 않음) |
| 생성자 탈퇴 | **가능**. 단, 워크스페이스는 유지된다. 관리 링크를 분실하면 설정 변경 불가 (MVP 한계) |
| 재참여 | 초대 링크가 유효하면 다시 참여 가능 (새 멤버로 취급) |

### 3-5. 워크스페이스 삭제

| 규칙 | 상세 |
|------|------|
| 삭제 | MVP에서는 **워크스페이스 삭제 기능 없음**. 사용하지 않으면 자연 소멸 |
| 정리 정책 | 90일간 활동 없는 워크스페이스는 비활성 처리 검토 (v2) |

## 4. 시간 규칙

### 4-1. 기본값

| 항목 | 기본값 |
|------|--------|
| 모집 시간 | 12:00 |
| 시간 선택 범위 | 11:00 ~ 14:00 (30분 단위) |
| 시간대 | 서버는 UTC 저장, 클라이언트는 **KST (Asia/Seoul)** 기준 표시 |

### 4-2. 마감 타이밍

| 규칙 | 상세 |
|------|------|
| 자동 마감 시점 | 모집 시간 정각 (예: 12:00 모집이면 12:00:00에 마감) |
| 마감 처리 방식 | 서버 크론잡 또는 요청 시점 체크. 1분 이내 지연 허용 |
| 마감 직전 참여 | 서버 시간 기준. 마감 시간 이후 요청은 거부 |

### 4-3. 날짜 규칙

| 규칙 | 상세 |
|------|------|
| 과거 날짜 모집 작성 | **불가** |
| 과거 날짜 모집 조회 | **가능**. 달력에서 과거 날짜 클릭하면 지난 모집글 확인 가능 |
| 미래 날짜 제한 | 최대 7일 후까지 (MVP). 너무 먼 미래는 의미 없음 |
| 날짜 기준 | 서버 시간 기준 KST 자정 (00:00)으로 날짜 구분 |

## 5. 알림 규칙 (MVP)

MVP에서는 **인앱 알림만** 지원한다. 푸시 알림은 v2.

### 5-1. 인앱 알림

MVP에서 인앱 알림도 최소화한다. 실시간 반영에 집중.

| 이벤트 | 알림 여부 | 대상 | 내용 |
|--------|----------|------|------|
| 새 모집글 작성 | X (MVP) | - | 메인 화면 새로고침/폴링으로 확인 |
| 참여자 추가 | X (MVP) | - | 상세 화면 실시간 반영 |
| 참여자 취소 | X (MVP) | - | 상세 화면 실시간 반영 |
| 모집 마감 | X (MVP) | - | 상태 뱃지로 표시 |
| 모집글 삭제 | X (MVP) | - | 메인 화면에서 사라짐 |

> **MVP 알림 전략**: 별도 알림 시스템 대신, 화면 데이터를 주기적으로 새로고침(폴링)하여 최신 상태를 반영한다. 알림 벨 아이콘 등은 v2에서 추가.

### 5-2. v2 알림 계획

| 이벤트 | 대상 | 내용 |
|--------|------|------|
| 새 모집글 | 워크스페이스 전체 멤버 | "{닉네임}님이 짬뽕 모집을 시작했습니다" |
| 참여자 추가 | 모집자 | "{닉네임}님이 참여했습니다 (3/4명)" |
| 참여자 취소 | 모집자 | "{닉네임}님이 참여를 취소했습니다 (2/4명)" |
| 인원 충족 마감 | 참여자 전원 | "짬뽕 모집이 마감되었습니다 (4/4명)" |
| 마감 임박 (30분 전) | 참여자 전원 | "짬뽕 모집 30분 전입니다" |
| 모집글 수정 | 참여자 전원 | "짬뽕 모집 정보가 수정되었습니다" |
| 모집글 삭제 | 참여자 전원 | "짬뽕 모집이 취소되었습니다" |

## 6. 데이터 보존 규칙

| 항목 | 규칙 |
|------|------|
| 지난 모집글 | 삭제하지 않음. 달력에서 과거 날짜 클릭하면 조회 가능 |
| 삭제된 모집글 | 소프트 딜리트. DB에 남아있지만 사용자에게 보이지 않음 |
| 참여 기록 | 삭제하지 않음. 히스토리로 활용 가능 (v2 통계) |
| 탈퇴한 멤버의 기록 | 남아있음. 닉네임은 "알 수 없는 멤버"로 표시하지 않고 당시 닉네임 유지 |

## 7. 엣지 케이스 정리

### 7-1. 모집자 관련

| 상황 | 처리 |
|------|------|
| 모집자가 모집글 삭제하면 참여자는? | 참여 자동 취소. 메인 화면에서 해당 모집글 사라짐 |
| 모집자가 워크스페이스를 탈퇴하면? | 기존 모집글은 유지 (작성자 닉네임 표시). 단, 수정/삭제/마감 불가 (토큰 소실) |
| 모집자의 브라우저 데이터가 삭제되면? | 새 멤버로 취급. 기존 모집글에 대한 수정/삭제 권한 소실 (MVP 한계) |

### 7-2. 참여자 관련

| 상황 | 처리 |
|------|------|
| 마감 후 참여 취소 요청 | 거부. "마감된 모집은 취소할 수 없습니다" |
| 동시 참여로 인원 초과 (race condition) | DB 레벨 트랜잭션으로 방지. 늦게 도착한 요청 거부 |
| 참여자가 브라우저 데이터 삭제 후 같은 모집에 다시 참여 | 새 멤버로 간주, 인원에 빈자리 있으면 참여 가능 (이전 참여 기록과는 별개 인물로 처리) |
| 인원이 꽉 찬 상태에서 누군가 취소하면? | 즉시 다시 "모집중"으로 전환 (인원 마감이 자동 마감 조건이었던 경우). 단, 수동 마감이나 시간 마감은 유지 |

### 7-3. 워크스페이스 관련

| 상황 | 처리 |
|------|------|
| 관리자(생성자)가 탈퇴하면? | 워크스페이스는 유지. 관리 링크를 아무도 모르면 설정 변경 불가 (MVP 한계) |
| 초대 링크가 외부에 유출되면? | 관리자가 링크 재발급으로 기존 링크 무효화. v2에서 가입 승인제 도입 검토 |
| 멤버 50명 도달 후 초대 링크 클릭 | "워크스페이스 멤버 수가 초과되었습니다" 메시지 |
| 같은 브라우저에서 같은 워크스페이스에 다른 닉네임으로 재참여 | 기존 토큰이 유효하면 재참여 불가 (이미 참여 상태). 쿠키 삭제 후 참여하면 새 멤버로 등록 |

### 7-4. 시간 관련

| 상황 | 처리 |
|------|------|
| 자정 직전에 작성한 모집글 (23:59) | 날짜는 작성 시점 기준. 시간 범위(11:00~14:00) 밖이므로 작성 자체가 불가 |
| 서버와 클라이언트 시간 차이 | 서버 시간 기준. 클라이언트는 서버로부터 현재 시간을 받아서 사용 |
| 여러 시간대의 멤버가 있으면? | MVP는 KST 고정. v2에서 워크스페이스 시간대 설정 검토 |

## 8. 보안 규칙 (MVP)

| 항목 | 규칙 |
|------|------|
| 사용자 식별 | httpOnly 쿠키에 저장된 토큰 (서버 발급, UUID) |
| 워크스페이스 접근 | 해당 워크스페이스의 유효한 멤버 토큰이 있어야 데이터 접근 가능 |
| API 인증 | 모든 API 요청에 쿠키 토큰 첨부. 유효하지 않은 토큰은 401 |
| 관리 기능 접근 | 관리 링크의 admin-token을 별도로 검증 |
| 초대 코드 | UUID v4 (추측 불가) |
| Rate limiting | 동일 IP에서 분당 60회 요청 제한 (모집 작성, 참여 등) |
| XSS 방지 | 모든 사용자 입력 (메뉴명, 장소, 메모, 닉네임) 서버에서 sanitize |

## 9. 성능 규칙

| 항목 | 목표 |
|------|------|
| 메인 화면 로딩 | 1초 이내 (FCP) |
| 참여 버튼 응답 | 500ms 이내 |
| 달력 월 전환 | 300ms 이내 |
| 동시 사용자 | 워크스페이스당 50명 기준 |
| 데이터 새로고침 | 30초 폴링 (메인 화면), 10초 폴링 (상세 화면) |
