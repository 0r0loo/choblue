# 비즈니스 규칙

---

## 1. 주문 규칙

### 1-1. 주문번호 체계

| 항목 | 규칙 |
|------|------|
| 형식 | `{접두사}-{순번}` (예: A-001, A-042) |
| 접두사 | 점주가 설정 (알파벳 1자, 기본 "A") |
| 순번 | 3자리 숫자, 001부터 시작 |
| 리셋 | 매일 자정(00:00) 001로 리셋 (설정 가능, 기본 ON) |
| 리셋 안 할 경우 | 999 이후 001로 순환 |

### 1-2. 주문 상태

```
[주문 생성] → 대기(PENDING) → 완료(COMPLETED)
                            → 취소(CANCELLED)
```

| 상태 | 설명 | 전이 가능 상태 |
|------|------|----------------|
| 대기 (PENDING) | 결제 완료 후 자동 생성 | 완료, 취소 |
| 완료 (COMPLETED) | 점주가 조리/전달 완료 처리 | 취소 (당일 한정) |
| 취소 (CANCELLED) | 점주가 취소 처리 (환불 포함) | 없음 (최종 상태) |

### 1-3. 주문 생성 조건

- 장바구니에 1개 이상의 상품이 있어야 함
- 모든 상품이 현재 판매 가능한 상태여야 함 (결제 직전 재검증)
- **최소 주문 금액: 없음** (v1에서는 제한하지 않음)
- **최대 주문 금액: 없음** (v1에서는 제한하지 않음)
- **최대 장바구니 항목 수: 50개** (실수로 과도하게 담는 것 방지)

### 1-4. 주문 데이터 보존

- 주문 생성 시점의 상품명, 가격, 옵션 정보를 스냅샷으로 저장
- 이후 상품이 수정/삭제되더라도 기존 주문 내역은 주문 당시 정보를 유지
- 주문 데이터 삭제: 없음 (영구 보존)

## 2. 결제 규칙

### 2-1. 결제 수단

| 항목 | 규칙 |
|------|------|
| v1 지원 | 카드 결제만 (토스페이먼츠) |
| 미지원 | 현금, 간편결제, 포인트 결제 |

### 2-2. 결제 프로세스

```
[결제 요청] → 토스페이먼츠 결제 위젯 → [카드 결제 승인]
           → 서버: 결제 승인 확인 (confirmPayment API)
           → 서버: 주문 생성
           → 클라이언트: 주문 완료 화면
```

**결제 승인과 주문 생성은 하나의 트랜잭션으로 처리한다.**
- 결제는 승인됐지만 주문 생성에 실패하면 → 결제 자동 취소(환불)
- 결제 취소도 실패하면 → 에러 로그 + 점주에게 수동 확인 알림

### 2-3. 결제 실패 처리

| 실패 유형 | 처리 |
|-----------|------|
| 카드 잔액 부족 | 에러 메시지 표시, 재시도 또는 취소 선택 |
| 카드 한도 초과 | 에러 메시지 표시, 재시도 또는 취소 선택 |
| 카드 오류 (유효기간 만료 등) | 에러 메시지 표시, 재시도 또는 취소 선택 |
| 네트워크 타임아웃 | 30초 대기 후 타임아웃 처리, 재시도 안내 |
| 토스페이먼츠 서버 오류 | "결제 시스템에 일시적 오류가 발생했습니다" 안내 |

### 2-4. 결제 취소 (환불)

| 항목 | 규칙 |
|------|------|
| 환불 가능 기간 | 결제일 당일 (자정 전까지) |
| 환불 방식 | 전액 환불만 지원 (부분 환불 미지원, v1) |
| 환불 주체 | 점주가 어드민에서 "주문 취소" 처리 |
| 환불 API | 토스페이먼츠 결제 취소 API 호출 |
| 환불 실패 시 | 에러 메시지 + "토스페이먼츠 대시보드에서 수동 환불" 안내 |

### 2-5. 토스페이먼츠 웹훅

- 결제 승인/취소 결과를 웹훅으로 수신
- 웹훅 수신 실패 시: 토스페이먼츠가 자동 재시도 (최대 10회)
- 웹훅과 API 응답이 불일치하면: 웹훅 기준으로 최종 상태 갱신
- 웹훅 서명(시크릿 키) 검증 필수

## 3. 상품 규칙

### 3-1. 상품 상태

| 상태 | 키오스크 표시 | 점주 어드민 | 설명 |
|------|-------------|------------|------|
| 판매중 (ACTIVE) | 정상 노출 | 목록 표시 | 기본 상태 |
| 품절 (SOLD_OUT) | 품절 오버레이, 선택 불가 | 목록 표시, "품절" 뱃지 | 재료 소진 시 |
| 숨김 (HIDDEN) | 노출 안 됨 | 목록 표시, "숨김" 뱃지 | 시즌 메뉴, 준비 중 |

### 3-2. 상품 삭제

- **소프트 삭제**: 주문 내역에 포함된 상품은 DB에서 실제 삭제하지 않음
- 삭제된 상품은 어드민 목록에서 제거, 키오스크에서도 제거
- 기존 주문 내역에서는 삭제된 상품의 정보가 스냅샷으로 유지됨

### 3-3. 옵션 규칙

| 항목 | 규칙 |
|------|------|
| 옵션 그룹 수 | 상품당 최대 5개 |
| 옵션 항목 수 | 그룹당 최대 10개 |
| 필수 옵션 | 반드시 1개 선택 (기본값 자동 선택) |
| 선택 옵션 | 0~N개 선택 (최대 선택 수 설정 가능) |
| 추가 금액 | 0원 이상. 음수(할인) 미지원 (v1) |

### 3-4. 가격 규칙

- 기본 가격: 0원 이상의 정수 (원 단위)
- 최종 가격 = 기본 가격 + 선택된 옵션 추가 금액의 합
- 장바구니 아이템 소계 = 최종 가격 x 수량
- 주문 총액 = 모든 아이템 소계의 합
- 가격 표시: 원화(₩) 또는 "원", 천 단위 콤마 (예: 5,500원)

### 3-5. 이미지 규칙

| 항목 | 규칙 |
|------|------|
| 형식 | JPG, PNG, WebP |
| 최대 크기 | 5MB |
| 이미지 수 | 상품당 최대 1장 (v1) |
| 이미지 없을 때 | 기본 플레이스홀더 이미지 표시 |
| 리사이징 | 업로드 시 서버에서 썸네일(200x200) + 원본(800x800) 생성 |

### 3-6. 카테고리 규칙

| 항목 | 규칙 |
|------|------|
| 카테고리 수 | 최대 20개 |
| 상품은 반드시 1개 카테고리에 소속 | 미소속 상품 불허 |
| 비활성 카테고리 | 키오스크에서 해당 카테고리 + 소속 상품 미표시 |
| 상품 없는 카테고리 | 키오스크에서 탭 미표시 |
| 삭제 조건 | 소속 상품 0개일 때만 삭제 가능 |

## 4. 영업 규칙

### 4-1. 영업시간

- 요일별로 개별 설정: 영업 여부 + 시작 시간 + 종료 시간
- 예: 월~금 09:00~21:00, 토 10:00~18:00, 일 휴무
- **자정 넘기는 영업**: v1에서는 미지원. 시작 시간 < 종료 시간 필수
- 영업시간 변경은 즉시 반영 (진행 중인 주문에는 영향 없음)

### 4-2. 영업시간 외 키오스크 동작

| 상황 | 키오스크 동작 |
|------|-------------|
| 영업시간 전 | 대기 화면에 "영업 준비 중입니다. 영업시간: HH:MM~HH:MM" 표시. 터치해도 메뉴 진입 불가 |
| 영업시간 종료 | 진행 중인 주문이 없으면 즉시 영업 종료 화면. 진행 중이면 해당 주문 완료 후 종료 |
| 휴무일 | "오늘은 휴무일입니다." 표시 |

### 4-3. 임시 휴무

- 점주가 어드민에서 "임시 휴무" 설정 가능
- 활성화 시 키오스크에 "임시 휴무입니다." 표시
- 진행 중인 주문이 있으면 경고 후 설정

## 5. 보안 및 권한 규칙

### 5-1. v1 권한 모델

| 역할 | 설명 |
|------|------|
| 점주 (OWNER) | 모든 어드민 기능 접근 가능. v1에서 유일한 어드민 사용자 유형 |
| 고객 | 키오스크만 사용. 로그인 불필요, 인증 없음 |

- v1에서 직원 계정은 지원하지 않음
- 한 매장 = 한 점주 계정
- 점주가 직접 가입 (이메일 + 비밀번호)

### 5-2. 인증 규칙

| 항목 | 규칙 |
|------|------|
| 인증 방식 | 이메일 + 비밀번호 (Better Auth) |
| 세션 유지 | JWT 기반, 7일 만료 |
| 세션 갱신 | 리프레시 토큰으로 자동 갱신 |
| 로그인 실패 | 5회 연속 실패 시 10분 잠금 |
| 비밀번호 규칙 | 최소 8자, 영문 + 숫자 포함 |
| 키오스크 인증 | 없음. 키오스크 앱은 매장 ID 기반으로 데이터 로드 (인증 불필요) |

### 5-3. API 보안

| 항목 | 규칙 |
|------|------|
| 어드민 API | 인증 필수 (JWT 토큰) |
| 키오스크 API | 인증 없음, 매장 ID 기반 조회 (공개 API) |
| 키오스크 API 범위 | 읽기: 카테고리, 상품, 매장 정보. 쓰기: 주문 생성만 |
| Rate Limiting | 키오스크 API: IP당 60 req/min. 어드민 API: 사용자당 120 req/min |

### 5-4. 키오스크 식별

- 키오스크 앱은 환경변수 또는 초기 설정으로 매장 ID를 가지고 있음
- 해당 매장 ID로 메뉴, 설정 등을 조회
- 키오스크 API는 인증 없이 접근 가능하지만, 주문 생성 시 매장 ID 검증

## 6. 엣지 케이스 및 예외 처리

### 6-1. 결제 중 네트워크 끊김

```
시나리오: 고객이 결제 버튼을 누른 후 네트워크가 끊어짐
```

| 단계 | 상황 | 처리 |
|------|------|------|
| 결제 요청 전 끊김 | 결제 API 호출 실패 | "네트워크 연결을 확인해 주세요." 메시지. 연결 복구 후 재시도 가능 |
| 결제 승인 요청 중 끊김 | 결제 상태 불확실 | 연결 복구 후 결제 상태 조회. 승인됐으면 주문 생성 진행. 미승인이면 재시도 안내 |
| 결제 승인됨 + 주문 생성 요청 중 끊김 | 결제는 됐지만 주문 미생성 | 연결 복구 후 결제 상태 확인 → 이미 승인된 결제로 주문 생성 재시도. 3회 실패 시 자동 환불 시도 |

**핵심 원칙**: 결제는 됐는데 주문이 안 만들어지는 상황을 절대 방치하지 않는다.
- 서버 측에서 미완료 결제를 주기적으로 스캔 (5분 간격)
- 승인 후 5분 내 주문 미생성 시 자동 환불 처리

### 6-2. 동시 주문 (키오스크 1대 기준)

- v1은 키오스크 1대이므로 물리적으로 동시 주문 불가
- 소프트웨어적 동시 요청 방지: 결제 버튼 중복 클릭 방지 (debounce + 버튼 비활성화)
- 주문번호 채번: DB 시퀀스 또는 원자적 카운터로 중복 방지

### 6-3. 주문 후 상품 품절

```
시나리오: 고객 A가 "딸기 스무디"를 장바구니에 담은 후,
         점주가 어드민에서 "딸기 스무디"를 품절 처리
```

| 시점 | 처리 |
|------|------|
| 장바구니에 담은 후, 결제 전 | 결제 시도 시 서버에서 상품 상태 재검증. 품절 상품이 있으면 결제 거부 + "일부 상품이 품절되었습니다. 장바구니를 확인해 주세요." 안내 |
| 결제 완료 후 | 이미 결제/주문 완료된 건은 유효. 점주가 해당 주문을 보고 고객에게 안내 후 필요 시 취소/환불 |

### 6-4. 토스페이먼츠 웹훅 실패

```
시나리오: 결제는 승인됐지만 웹훅이 서버에 도달하지 못함
```

| 처리 | 상세 |
|------|------|
| 1차 방어 | 클라이언트 → 서버로 결제 승인 결과 직접 전달 (redirect 방식). 웹훅 없이도 주문 생성 가능 |
| 2차 방어 | 토스페이먼츠 웹훅 자동 재시도 (최대 10회, 지수 백오프) |
| 3차 방어 | 서버 배치 작업: 미처리 결제를 토스페이먼츠 조회 API로 확인 (5분 간격) |
| 최종 불일치 | 서버 DB 결제 상태 vs 토스페이먼츠 상태 불일치 시 토스페이먼츠 기준으로 보정 |

### 6-5. 서버 다운 시 키오스크 동작

| 상황 | 처리 |
|------|------|
| 초기 로딩 실패 (메뉴 못 불러옴) | "서비스 준비 중입니다. 잠시 후 다시 시도해 주세요." 화면. 30초마다 자동 재시도 |
| 주문 중 서버 응답 없음 | 각 API 호출에 10초 타임아웃. 타임아웃 시 "일시적 오류" 안내 + 재시도 버튼 |
| 결제 후 주문 생성 API 실패 | 6-1 결제 중 네트워크 끊김 규칙과 동일하게 처리 |

### 6-6. 데이터 정합성

| 규칙 | 상세 |
|------|------|
| 가격 스냅샷 | 주문 생성 시 상품/옵션 가격을 주문 테이블에 복사. 이후 가격 변경해도 기존 주문 불변 |
| 결제 금액 검증 | 서버에서 장바구니 아이템 기준으로 금액 재계산 후 결제 요청 금액과 비교. 불일치 시 결제 거부 |
| 주문번호 유일성 | DB unique constraint. 충돌 시 자동 증가 후 재시도 |

### 6-7. 키오스크 장시간 방치

| 상황 | 처리 |
|------|------|

### 6-8. 브라우저/앱 크래시

- 키오스크 앱이 크래시 후 재시작되면 대기 화면부터 시작
- 로컬 스토리지의 장바구니 데이터는 세션 단위로 관리 (크래시 시 소멸)
- 진행 중이던 결제가 있으면 서버 배치 작업에서 정리 (6-4 참고)
